## 四、关键发现与分析

### 4.1 最优质量：exp06（late_sem + stopdensify15000）
- **PSNR 最高**（25.04 dB），**SSIM 最高**（0.702），**LPIPS 最低**（0.287），说明在延迟启动语义 loss（5000 迭代）并将 densification 进行到 15000 迭代的策略下，模型在渲染质量和语义精度上达到了最佳平衡。
- 但代价是模型规模最大（Avg Visible Count ≈ 474k），推理速度（92.7 FPS）略低于无 densify 的方案。

### 4.2 最佳速度：exp03_scratch（从头训练，无 densify）
- **推理速度最快**（143.1 FPS），且模型最小（≈ 175k 可见高斯）。
- 然而质量最差：PSNR 仅 24.41，LPIPS 高达 0.388，mIoU 也最低。**不推荐用于质量敏感场景**，但可作为实时性要求极高的基线。

### 4.3 质量与速度的最佳折中：exp03_nodensify（预训练，无 densify）
- 在**预训练权重基础上关闭 densify**，质量与 exp01/exp02 几乎持平（PSNR 24.63，SSIM 0.687），但**推理速度翻倍**（85.2 FPS vs 29~46 FPS）。
- 模型大小也略有缩减（≈ 386k）。这是一个非常实用的**高效推理方案**。

### 4.4 Densification 停止时机的影响（exp04 ~ exp06 对比）
- `update_until` 越早停止，模型越小、速度越快，但质量呈**明显下降趋势**：
  - 5000 停止 → PSNR 24.79, FPS 118.8（质量损失较大，LPIPS 0.342）
  - 10000 停止 → PSNR 24.87, FPS 105.3（接近最优）
  - 15000 停止 → PSNR 25.04, FPS 92.7（质量最优）
- 结论：**densification 持续到 10000~15000 迭代对最终质量有显著正向作用**，但 15000 之后的边际收益递减。

### 4.5 语义 Loss 启动时机（exp01 vs exp02）
- exp01 从第 0 迭代启动语义 loss 并引入 mask warmup/ramp；exp02 延迟到 5000 迭代启动。
- 两者最终指标几乎相同（PSNR 24.70 vs 24.64），但 exp01 的推理速度更慢（29.2 vs 46.4 FPS）。
- 结论：**过早启动语义 loss 会显著降低推理性能**，而延迟到 5000 迭代启动可以在不牺牲质量的前提下提升约 **58% 的 FPS**。

### 4.6 Focal Loss 参数的影响
- exp01/exp02 使用 `focal_alpha=0.75`；exp03~exp06 使用 `focal_alpha=0.25`。
- 从 exp02 到 exp03（同为预训练+延迟语义），降低 `focal_alpha` 并未带来质量提升，反而使 PSNR 微降（24.64 → 24.63），但速度大幅提升（46.4 → 85.2 FPS），这主要归因于 `update_until=0`（关闭 densify）而非 focal_alpha 本身。

### 7.3 twostage 结果

| 指标 | ITER 5000 Test | ITER 7000 Test | ITER 10000 Test | ITER 10000 Train | 最终渲染 |
|------|---|---------------|---------------|----------------|------------------|----------|
| PSNR ↑ | **25.04** | **25.04** | **25.04** | 27.75 | 25.04 |

| LPIPS ↓ | - | - | - | - | 0.2870 |
| FPS ↑ | - | - | - | - | 91.8 |

> **分析**：Two-Stage 训练是 Phase 3 **唯一成功的改进**。仅加载 exp06 的预训练几何并冻结所有参数，用 10000 迭代训练深层 SegMLP，即可在**不损失任何 RGB 质量**的前提下达到与 exp06 完全相同的语义精度（mIoU 0.7776 ≈ exp06 的 0.7778）。
> - PSNR/SSIM/LPIPS 与 exp06 几乎完全一致（25.04 / 0.702 / 0.287）
> - 推理速度也与 exp06 相当（91.8 vs 92.7 FPS）
> - 训练时间大幅缩短（只需优化 ~55K 参数）
> 
> **关键洞察**：当几何已经充分收敛后，语义分割可以作为一个完全独立的第二阶段来训练。这避免了 seg loss 对 RGB 重建的干扰，同时深层 MLP 有足够的能力拟合语义标签。**强烈推荐此方案**用于已有高质量几何的场景。

> ⚠️ **注意**：在修复 `resolution=2` 之前，曾有一次错误配置的运行（`resolution=-1`，全分辨率），其 PSNR 仅 23.13，mIoU 0.77。该结果因分辨率不一致而被弃用。本节数据均来自正确配置 `--resolution 2` 的运行。

### 7.4 per_gaussian 结果

| 指标 | ITER 7000 Test | ITER 30000 Test | ITER 30000 Train | 最终渲染 |
|------|---|---------------|-----------------|------------------|----------|
| PSNR ↑ | 23.98 | **24.78** | 26.30 | 24.78 |
| mIoU ↑ | 0.5767 | **0.4264** | 0.4653 | - |
| SSIM ↑ | - | - | - | 0.6855 |
| LPIPS ↓ | - | - | - | 0.3210 |
| FPS ↑ | - | - | - | 59.2 |

> **分析**：per-Gaussian 分割（每个 offset 独立 seg 值）表现**非常差**。mIoU 从 7000 迭代的 0.58 进一步**下降到** 30000 迭代的 0.43，说明模型完全没有学到有意义的分割语义。可能的原因：
> 1. `n_offsets=10` 个值共享同一个 anchor feature，但每个值独立预测，特征空间不足以支撑 10 个独立且语义一致的输出；
> 2. KNN loss 是在 anchor 级别做的（对 offsets 取平均），无法约束 per-Gaussian 值的空间一致性；
> 3. mask rendering 的 alpha blending 对 per-Gaussian seg 值非常敏感，微小差异会被放大。
> 此方案**完全不推荐**。

### 7.5 Phase 3 综合对比

| 实验 | Test PSNR ↑ | Test mIoU ↑ | SSIM ↑ | LPIPS ↓ | FPS ↑ | 结论 |
|------|---|------------|-------------|--------|---------|-------|------|
| **baseline (exp06)** | **25.04** | **0.7778** | **0.7017** | **0.2870** | 92.7 | 最优基线 |
| v2_full_30000 | 24.43 | 0.6065 | 0.6653 | 0.3486 | 66.2 | ❌ 显著劣于基线 |
| **twostage** | **25.04** | **0.7776** | **0.7017** | **0.2870** | 91.8 | ✅ **达到基线，训练更快** |
| per_gaussian | 24.78 | 0.4264 | 0.6855 | 0.3210 | 59.2 | ❌ 语义完全失效 |

### 7.6 关键结论

1. **Two-Stage 训练是最佳实践**：先训练高质量几何，再冻结并单独优化深层 seg head，可在不牺牲 RGB 的前提下达到相同的 mIoU，且训练成本大幅降低。
2. **深层 SegMLP 单独使用无益**：v2_full 从头训练深层 MLP 反而降低了 mIoU（0.61 vs 0.78），说明深层 seg head 的优势只有在**几何已固定**时才能发挥。
3. **per-Gaussian seg 不可行**：独立预测每个 offset 的 seg 值导致语义完全崩溃，mIoU 仅 0.43。锚点级别的共享语义是 Scaffold-GS 架构的必要设计。

   - COB-GS 在 myvideo 上 mIoU 仅 **63.62%**，显著低于 Scaffold-GSLFY 最佳（80.22%），差距 **16.6 个百分点**。

#### 4. 二分类数据误用 num_classes>1 的影响分析

**训练阶段**:
- GT 只有 0/255，若设 num_classes=20:
  - CE loss: 模型被迫在 20 个 channel 上做 softmax，19 个类别永无正样本
  - Dice loss: 不存在的类别 pred_probs > 0 但 target=0，产生错误惩罚梯度
  - 通道 0 概率被稀释 → sigmoid threshold 0.5 失效

**评估阶段**:
- eval_myvideo.py 对 num_classes>1 使用 `pred_mask.argmax(dim=0)`
- 只有 class 0 有 GT 支持（255 被 ignore）
- Classes 1-19: IoU = 0
- mIoU = IoU_0 / 20 ≈ 5% 真实值

**eval_myvideo.py 自动检测** (line 257-258):
```python
if args.num_classes == 1 and cfg_num_classes > 1:
    num_classes = cfg_num_classes
```
- 若用户显式传 `--num_classes 1`，但 cfg_args 中 num_classes=20，eval 会**自动覆盖为 20**
- 这导致训练和评估语义不一致（训练用 1，评估用 20）
- **这是严重 bug**: 用户无法通过显式传参强制使用二分类评估

**结论**: Claude 判断 **正确**。二分类数据误用 num_classes>1 会导致训练和评估双重崩溃。

### 5. 自适应训练参数工具：`scripts/prescan_adaptive.py`

#### 5.1 功能
预扫描数据集 mask，根据统计特征自动推荐训练超参数。处理极端类别不平衡的三个策略：

1. **类别合并（Class Merging）**：将出现 <10 张图或像素占比 <0.05% 的类合并为 "other"
2. **类别权重（Loss Weighting）**：支持 inverse_sqrt、effective_number、inverse_freq 三种策略，clamp [0.1, 3.0]，归一化后均值=1.0
3. **过采样建议（Oversampling）**：识别包含稀有类的图片，建议 2-3x 过采样

## 2026-05-30 Scheduled Joint Training 实验（mask_weight decay + opacity gradient gating）

### 背景

在发现 `no_opacity_detach` 是突破性改进（dronev4_2 mIoU 91.65%）后，进一步提出两项训练调度策略：
1. **Mask Weight Decay**：mask_weight 从高值线性衰减到低值，早期强语义、后期保护 RGB
2. **Opacity Gradient Gating**：分阶段控制 mask→opacity 梯度，早期允许、后期截断

### 创新点

#### 创新1：Mask Weight Decay Schedule
- **原理**：语义 mask 损失权重不应该是常数。训练早期需要较高 mask_weight 快速学习分割，后期应降低以减少对 RGB 的干扰
- **实现**：线性衰减插值，在 `mask_decay_start` 到 `mask_decay_end` 之间从 `mask_weight` 线性衰减到 `mask_weight_final`
- **公式**：`seg_w = seg_w + t * (mask_weight_final - seg_w)`，其中 `t = (iter - start) / (end - start)`
- **新增参数**：`--mask_weight_final`, `--mask_decay_start`, `--mask_decay_end`

#### 创新2：Opacity Gradient Gating
- **原理**：mask 损失通过 opacity MLP 反传的梯度在不同训练阶段应有不同策略
  - 训练早期（几何未稳定）：允许 mask→opacity 梯度流动
  - 训练后期（几何已收敛）：截断该梯度，防止持续干扰
- **实现**：`--no_opacity_detach` + `--opacity_grad_until <iter>`
- **关键设计**：必须同时满足 `no_opacity_detach=True` AND `opacity_grad_until > 0` 才启用门控，保证向后兼容
- **新增参数**：`--opacity_grad_until`

### 实验配置

Train/test split：每 8 张取 1 张作测试（`--eval`）

### 失败原因分析

1. **场景尺度太大**：spatial_lr_scale=463.6 米，默认参数为小场景（~10 米）设计。
2. **offset_lr 相对场景过小**：实际 offset_lr=0.046 米，在 463 米场景中几乎无法移动，Gaussian 分布僵化。
3. **scale clamp 过严**：`clamp_max(dist2, 4.0)` 限制初始 scale ≤ 2 米，但 100 万点分布在 463 米范围平均间距 3-4 米，大量空隙无法覆盖 → 渲染图像大面积背景色。
4. **position_lr=0**：anchor 位置完全固定，仅靠 offset 微调，大场景下无法自适应。
5. **Loss=nan 持续存在**：EMA 被早期 NaN 永久污染，但 mask loss 正常，说明 NaN 来自 RGB 通道（feat 梯度被污染，mask 用 detach 不受影响）。

## uav_HKisland_GNSS03 Loss=NaN 根因分析与修复（2026-06-05）

### 问题现象
- 训练从第 0/第 5 个 iteration 开始 `Loss=nan`，持续至训练结束
- `--detect_anomaly` 未捕获梯度异常，说明 NaN 来自前向传播而非梯度爆炸
- 之前尝试的所有参数调整（voxel_size、offset_lr、ratio、dist2 clamp）均无效

### 根因定位
1. **空 tensor 导致 `scaling_reg=NaN`**
   - `scaling_reg = scaling.prod(dim=1).mean()` 在 `scaling` 为空 tensor 时返回 NaN
   - `scaling` 为空是因为 `prefilter_voxel()` 返回了全 False（没有可见 anchor）

2. **97/467 个 camera 完全没有可见 anchor**
   - Camera centers range: x=[-381.9,410.7], y=[-353.5,411.6], z=[-81.0,143.8]
   - Anchor range:       x=[-251.4,108.0], y=[-529.8,140.3], z=[-130.9,23.2]
   - 大量 camera 位于 anchor 覆盖范围之外（如 x=410, y=292 等），这些 camera 看向场景时视锥内无任何 anchor
   - 当随机选中这些 camera 时，渲染全黑 → `scaling` 为空 → `scaling_reg=NaN` → `loss=NaN`

3. **初始 scale 过大（辅助因素）**
   - spatial_lr_scale=463.6 时，`distCUDA2` 最大距离达 60m
   - `exp(4.1) ≈ 60m` 的初始 scale 可能触发 rasterizer 数值问题
   - 已添加 `clamp_max(..., 2.5)` 限制初始 scale 上限为 `exp(2.5) ≈ 12.2m`

### 修复方案

| 文件 | 修改 | 说明 |
|------|------|------|
| `train.py` | `if scaling.numel()>0: scaling_reg=... else: scaling_reg=0` | 空 tensor 时跳过 scaling_reg，避免 NaN |
| `scene/gaussian_model.py:362` | `scales = clamp_max(log(sqrt(dist2)), 2.5)` | 限制初始 scale 上限，防止超大高斯 |

### 修复后验证

```bash
python train.py -s data/uav_HKisland_GNSS03 -m outputs/uav_HKisland_GNSS03_fix_empty \
  --num_classes 8 --voxel_size 0.05 --resolution 1 --start_semantic_iter 10000 \
  --update_until 20000 --mask_weight 0.4 --knn_weight 0.05 --focal_alpha 0.25 \
  --focal_gamma 2.0 --knn_every 100 --knn_offset 55 --no_opacity_detach \
  --eval --iterations 500 --port 0 --offset_lr_init 0.0003 --ratio 2 \
  --densify_grad_threshold 0.0001
```

- **Loss 不再 NaN**：iter 0~500 保持 0.30~0.46 正常范围
- **[ITER 500] Test PSNR 11.40 dB, mIoU 0.0400**（语义 loss 尚未启动，mIoU 低为预期）
- **[ITER 500] Train PSNR 13.51 dB**

### 遗留问题
- 97/467 (21%) 的 training camera 在场景外，这些 iteration 对训练无贡献（渲染全黑，无梯度）
- 长期方案：应在数据预处理阶段过滤掉这些 camera，或调整 COLMAP 重建范围

   - **两者 PSNR 不可直接对比**，但 human-mask 内部的相对排名有意义

4. **排名一致性**：
   - human-mask 上：**exp02 > exp01 > exp06 > nodetach**
   - 与早期 myvideo 评估（2026-05-06）趋势基本一致（exp02 领先）
   - nodetach 在 human-mask 上不再是绝对第一，说明其突破性的 91% mIoU（pseudo-mask）不能简单迁移到真实人工标注

### 结论

- **以后所有涉及 "人工评估" 的结果必须明确标注 GT 来源**（`masks_human/` 或 `masks/`）
- `eval_heldout_semantic.py` 的结果应称为 **"pseudo-mask held-out evaluation"**，而非人工评估
- `eval_human_only.py` 的结果才是 **"human-annotated evaluation"**
- 论文中若报告 dronev4_2 的人工标注精度，应使用本节的 37 张 human-mask 结果（最佳 FG IoU ≈ 0.625）
